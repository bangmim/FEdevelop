## Q.1 React의 주요 특징

## A.1

1. 선언형 프로그래밍
2. 컴포넌트 기반 아키텍처
   HTML,CSS,JS를 하나로 묶어 개발
   각 컴포넌트는 독립적이고 재사용 가능하며, 여러 곳에서 사용할 수 잇음
3. Virtual DOM
   실제 DOM을 추상화한 개념, 메모리 내에 존재하는 DOM
4. 단방향 데이터 바인딩
   데이터가 부모 컴포넌트에서 자식 컴포넌트로 한 방향으로 흐름
   데이터 흐름이 명확하여 예측 가능성이 높아지며 단순하여 이해하기 쉬움

## Q.2 Dom과 Virtual Dom의 차이점

## A.2

실제 돔은 HTML마크업을 구조화한 것
버츄얼 돔은 UI업데이트를 효율적으로 하기 위해 사용
버츄얼 돔은 실제 돔을 추상화
이전 버츄얼 돔과 새로운 버츄얼 돔을 비교하여 실제로 변경한 부분을 돔에 적용

## Q.3 Props와 State 차이점

## A.3

Props는 부모에서 자식 컴포넌트로 전달하는 읽기 전용 데이터
직접 변경 불가능, 변경하려면 부모 컴포넌트에서 변경 또는 업데이트 함수 호출
State는 컴포넌트 내부에서 관리하는 동적 데이터, 컴포넌트 내부에서 수정 가능

## Q.4 useEffect는 어떻게 작동하나요?

## A.4

함수형 컴포넌트에서 사이드 이펙트를 수행하기 위해 사용됩니다. 컴포넌트가 렌더링 될때와 의존성 배열이 변경될 때마다 특정 작업을 수행할 수 있습니다.

## Q.5. 전역 상태를 사용하는 경우는 언제일까?

## A.5

1. 여러 컴포넌트에서 동일한 데이터가 필요할 때
   ex. 사용자 인증정보
2. 상위 컴포넌트에서 하위 컴포넌트로 깊게 상태를 전달할 때
   전역 상태 없다면 Prop Drilling(중간에 위치한 컴포넌트들이 필요 없는 프롭들을 전달해야하는 상황 )로만 해야함
3. 캐시 또는 상태 동기화가 필요한 경우

전역상태가 과도하면 상태 관리가 복잡해지기 때문에 적절한 사용이 필요하다.

## Q.6. 상태관리 라이브러리를 사용하는 이유?

## A.6

리액트에서는 ContextAPI 가지고있음
프로젝트가 커지면 관리가 어려움
한계를 보완하기 위해 Redux, zustand가 나옴

한계 : 전역적 상태 공유지만 단순한 상태관리에 적합. 상태가 복잡해지거나 비동기가 많이 발생하는 경우 한계 발생, 불필요한 리랜더링, 상태 분리 및 확장성 문제
context가 중첩으로 사용하게 됨 > 복잡
작은 규모 애플리케이션에 적합

필요한 컴포넌트만 리랜더링 되도록 > 컴포넌트 성능 최적화 ( 상태 변경이 빈번한 대규모 어플리케이션에서 성능 유지에 도움이 됨)
상태 로직 분리와 유지보수성 향상 : 중앙에서 상태 관리.
테스트 용이성 : 컴포넌트를 배제하고 함수만 따로 테스트가 가능하다

## Q.7 Redux의 3대 원칙

## A.7

1. Single Source of Truth : 일관성 문제 해결
2. State is Read-Only : 상태는 오직 읽을수만 있고, 수정(데이터 변경)은 액션을 이용해서만 가능하다 > 상태 변화 추적 가능
3. Reducers are Pure Functions : 순수함수란 동일한 입력이 주어졌을 때 항상 동일한 출력을 내고 외부 상태나 값에 영향받지 않음. 상태가 변경되는 모든 로직은 리듀서 안에서 이루어짐. 상태 변화를 한눈에 파악 가능하다.

## Q.8 React에서 컴포넌트를 어떻게 구성하는게 효율적일까?

## A.8

리액트의 장점은 컴포넌트 기반의 아키텍처라는것. 효율적으로 구성하지 않으면 유지보수와 확장에 어려움을 겪을 수 있음

- Atomic Design 및 컴포넌트 계층 구조 : 컴포넌트를 작은 단위에서 큰 단위로 나누는 방법
  버튼:기본요소 원자적 컴포넌트 -> 폼, 카드 : 몰레큘러 컴포넌트 -> 페이지 : 올가니즘 컴포넌트
- 단일 책임 원칙
- 상태 관리와 데이터 흐름
- Stateless vs. Stateful 컴포넌트 : 적절하게 분리하는 것이 필요
  Stateless - 상태를 가지지않고 주로 데이터를 받아서 UI를 랜더링하는 역할. 컴포넌트 자체가 재사용성이 높아지고 테스트가 용이해짐
  Stateful - 상태를 관리하는 컴포넌트. 주로 부모 컴포넌트

## Q.9 Context API 와 Redux(외부 상태관리 라이브러리)비교

## A.9

- 리덕스 : 상태를 중앙 저장소에 저장하고 상태를 변경하는 방법은 액션을 디스패치 하는것.
  코드의 양이 굉장히 방대하게 많고, 불필요한 보일러플레이트가 굉장히 많다.
  하지만 그만큼 대규모 애플리케이션에서는 상태 변경이 명확하게 보이기 때문에 유지보수가 쉬워진다는 장점을 가진 라이브러리.
  비동기 액션에서 더 체계적으로 사용할 수 있다.
  미들웨어를 사용하여 체계적으로 관리
  복잡성이 높음
  큰 규모의 프로젝트

## Q.10 hook은 무엇일까?

## A.10

클래스형 컴포넌트 : 라이프 사이클, 랜더
hooks: useState, useEffect, useContext, useReducer 등
훅형 컴포넌트 : 간결성과 재사용성. 직관적으로 로직 분리 가능
여러 컴포넌트에서 커스텀 훅을 사용하여 로직을 재사용할 수 있도록 구성
클래스 컴포넌트의 복잡성 해결. 함수형 컴포넌트에서 상태와 라이프사이클 주기를 쉽게 관리할 수 있도록 도와주는 강력한 도구
ex. useFetch를 이용하여 api를 주고받는 훅을 만들어 다양한 곳에서 사용.
